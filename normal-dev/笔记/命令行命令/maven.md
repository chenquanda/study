### 一、 核心生命周期命令（最常用）
Maven 有三套独立的生命周期，我们最常用的是 `default` 生命周期，它的各个阶段是有序的，执行后面的命令会自动把前面的命令也执行一遍。
| 命令          | 作用说明   | 实际发生了什么                                               |
| :------------ | :--------- | :----------------------------------------------------------- |
| `mvn clean`   | 清理       | 删除 `target` 目录（清空之前编译和打包生成的旧文件）。       |
| `mvn compile` | 编译       | 只编译项目的源代码（`src/main/java`），不编译测试代码，不打包。 |
| `mvn test`    | 测试       | 执行项目中的单元测试代码（`src/test/java`）。*(会先执行 compile)* |
| `mvn package` | 打包       | 将编译好的代码打包成 jar 包或 war 包，存放在 `target` 目录下。*(会先执行 test)* |
| `mvn install` | 安装到本地 | 将打包好的 jar/war 安装到你的本地 Maven 仓库中，供其他本地项目引用。*(会先执行 package)* |
| `mvn deploy`  | 发布到远程 | 将打包好的文件发布到私服（如 Nexus）或中央仓库，供全团队使用。*(会先执行 install)* |
**🔥 黄金组合命令：**
在做项目部署或提交代码前，最常用的组合是：

```bash
# 先清理旧的，再打包（跳过测试以节省时间）
mvn clean package -DskipTests
```
或者
```bash
# 先清理，再安装到本地仓库（通常用于多模块项目，比如自己写了一个通用模块供其他模块依赖）
mvn clean install -DskipTests
```

### 二、 依赖管理命令
当你遇到依赖冲突、想知道项目到底引入了哪些 jar 包时，使用以下命令。
| 命令                  | 作用说明                                                     |
| :-------------------- | :----------------------------------------------------------- |
| `mvn dependency:tree` | **查看依赖树**。非常实用！能清晰看到项目引入了哪些依赖，以及它们之间的层级（父子）关系，用于排查依赖冲突。 |
| `mvn dependency:list` | 以平铺的列表形式查看所有依赖。                               |

### 三、 跳过测试的参数（附加在命令后面）
测试代码如果连接了数据库或很耗时，打包时通常会选择跳过。
| 参数                     | 作用说明                                                     |
| :----------------------- | :----------------------------------------------------------- |
| `-DskipTests`            | **推荐**。不执行测试用例，但**依然会编译**测试代码。         |
| `-Dmaven.test.skip=true` | 绝对跳过。**不编译**测试代码，也不执行测试用例。（速度最快，但不规范） |
**使用示例：**
```bash
mvn clean package -DskipTests
```

### 四、 多模块项目命令
如果你的项目是由多个子模块组成的（父工程下有多个子目录，每个子目录都有 `pom.xml`）：
| 命令                                 | 作用说明                                                     |
| :----------------------------------- | :----------------------------------------------------------- |
| `mvn clean install -pl moduleA`      | **只构建指定的模块**（moduleA），不构建其他模块。            |
| `mvn clean install -pl moduleA -am`  | 构建指定模块（moduleA），**并且同时构建它所依赖的模块**（Also Make）。 |
| `mvn clean install -pl moduleA -amd` | 构建指定模块（moduleA），**并且同时构建依赖它的下游模块**（Also Make Dependents）。 |

### 五、 问题排查与高级命令
| 命令                     | 作用说明                                                     |
| :----------------------- | :----------------------------------------------------------- |
| `mvn help:effective-pom` | 查看有效的 POM。当你不确定父 POM 或继承的配置最终合并成什么样时，用这个命令可以看到最终生效的完整 `pom.xml`。 |
| `mvn archetype:generate` | **创建项目**。通过交互式命令从零开始生成一个标准的 Maven 骨架项目。（实际开发中通常直接用 IDEA 创建，很少手敲这个命令）。 |
| `mvn -U`                 | **强制更新**。加在其他命令后面（如 `mvn clean install -U`），强制让 Maven 去远程仓库拉取最新的依赖快照版（SNAPSHOT）和插件，不使用本地缓存。 |

### 💡 遇到报错时的排查思路：
如果在执行 `package` 或 `install` 时报错，请按以下顺序看日志：
1. 看 **红色报错日志的最上面几行**（Maven 的根本原因通常在最上面，下面的一堆红色通常是连锁反应）。
2. 检查是否是 **依赖下载失败**（通常提示 `Could not resolve dependencies`）。解决：检查网络，或者检查 `settings.xml` 里的阿里云镜像配置是否写错。
3. 检查是否是 **编译报错**（提示 `Failed to execute goal org.apache.maven.plugins:maven-compiler-plugin`）。解决：通常是代码有语法错误，或者 JDK 版本不对（检查 `pom.xml` 中的 `<java.version>` 或 `<maven.compiler.source>`）。
